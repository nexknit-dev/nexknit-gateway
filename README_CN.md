## **NexKnit —— 由真实故事开始的回答**


硕士期间，实验室的服务器被人开了一个端口。随后它被挂到云端，被用来通过 VSCode.Dev 访问并进行远程开发。这很 cool，并且是良好的远程开发实践，但是它到底还是开放了公网端口，最终导致服务器被骇入。这指向了一个痛点：安全的随时随地的看一眼任意一个能联网的设备上的性能和状态。

它催生了我们的 NexKnit，一个专门为个人安全地监控能联网节点上任何状态而生的仪表盘。

- 在任意一个能联网的节点上向网关发送 `Name | Type | Info` 格式的文本即可在仪表盘上查看。
- 无须考虑任何安全问题，只通过 HTTPs 向云端推送数据，而推送的返回值只会被丢弃。
- 服务器上只需要部署一个仅依赖 Python3.9+及官方库的网关。没有依赖，极易审查。
- 完全免费，7*24小时的四节点监控能力。足以驾驭Homelab或者小规模AI训练。
- 快速集成，通过TCP监听环路，没有语言限制，你甚至可以通过curl发送数据。
- 极易部署，三分钟内即可完整部署流程，并且无须调整节点配置。

## 🏛️ 架构设计与特性

<details>
<summary><b>易用</b> — 零环境侵入，极易集成</summary>

我们希望 nexknit 不改造用户的环境。很多监控方案需要装一堆依赖、配 YAML、启动守护进程，这对只想看一眼训练进度的 AI 学生来说太重了。

所以我们定了一个硬约束：网关只依赖 Python 自带的库。任何装了 Python 3.9+ 的机器，`python main.py` 就能跑，不用 `pip install`。

更关键的一点：我们不让网关侵入业务，而是让业务侵入网关。网关和采集器之间用纯文本的 TCP 协议通信。任何能往本地端口写一行文本的东西——Shell 脚本、C 程序、MQTT 客户端、Bash——都是合法的采集器。它不需要导入任何库，不需要理解网关的内部实现，只需要发一行 `I|CpuTemp|65.3`。这行文本就是一个完整的数据上报。

你不需要读 SDK 文档，不需要管库版本，不需要担心语言兼容性。你的程序能往本地端口写一行文本，它就已经是一个完整的 nexknit 采集器。

</details>

<details>
<summary><b>安全</b> — 零入站、零攻击面</summary>

我们的安全设计只有一个出发点：暴露公网端口是不可接受的。穿透和端口转发说到底是在网络边界上开门，无论后续怎么加固，攻击面已经扩大了。

所以 nexknit 不监听任何对外端口。网关只做一件事：主动往 Cloudflare Worker 发一个短暂的 HTTPS 请求，把数据推上去，然后立刻关掉连接。Worker 返回的响应会被直接丢弃，网关不做任何处理。

这形成了一个物理级别的单向阀——信息只能从内网流向公网，公网没有任何途径与内网网关建立连接。这是我们说的“零入站、零攻击面”。

关于延迟双工：我们注意到有人可能想从外部往节点发一些简单的控制指令，比如触发一次采集、调整频率。这在不开入站端口的前提下是可以做的——网关在每次推送的时候顺带拉一个“待执行指令队列”就行。但这个特性涉及额外的安全设计和免费额度消耗评估，我们还没把它放进正式开发计划。如果你有明确的需求或想法，欢迎在 Issue 里讨论。如果需求足够明确，我们会考虑加速推进。

</details>

<details>
<summary><b>伪流畅</b> — Jitter Buffer 平滑展示</summary>

公网丢包是常态。我们实测过，跨国链路的随机丢包率能到十几个百分点。如果直接把拉到的数据渲染到仪表盘上，折线图会经常断、状态灯会莫名闪。这对一个正在焦虑等训练结果的人来说很难接受。

但我们不能用加请求或建长连接来保证实时性——免费额度摆在那里。所以我们换了个思路：数据到达后不立刻展示，先放进一个缓冲区。仪表盘里有一个虚拟时钟，按原始时间戳的顺序逐点把数据推出来。如果中间某个时间点的数据还没到（因为它还在公网上漂着），时钟就跳过它，不影响后面的点。

你看到的图是连续平滑的。那些迟到或丢失的点在背后被处理掉了。这不是真的流畅，是在不可靠的传输层之上构造出来的稳定视图。我们管它叫“伪流畅”。

</details>

<details>
<summary><b>可靠抵达</b> — nexus 滑动窗口算法</summary>

这是 nexknit 对可靠性问题的回答。我们面对的约束是：零入站（不能用 ACK）、短连接（不能用 TCP 重传）、免费额度（不能增加请求次数）。传统的可靠性手段在这里全部失效。

剩下的唯一路径是应用层冗余。我们不是“发现丢了再重发”，而是“提前多给，宁可重复，绝不遗漏”。每次推送不只带当前批次的数据，还附带前两次的旧批次。一个数据包会在三次独立的推送窗口里被重复携带。

在 5% 的随机丢包率下，三次都丢的概率是 0.05³ = 0.0125%。我们用超过一万个真实网络包验证过这个模型。具体的压测数据和存活率，见“设计模型与实测数据”章节。

</details>

<details>
<summary><b>面向免费额度设计</b> — 永久免费，无需信用卡</summary>

免费是 nexknit 的基石。我们选 Cloudflare 作为首选，因为它目前给个人开发者提供了最慷慨的免费计划——Worker 请求次数和 D1 存储空间足够用。

但 nexknit 的云端部分只做透传，和 Cloudflare 没有深度绑定。如果将来政策变了，或者有更好的免费服务商出现，只需要改极少量的云端代码就能迁走。可能会因为不同平台的免费额度差异导致一些降级，但不会跑不了。

我们刻意只用短连接。不是因为长连接不好，是因为几乎所有云厂商的免费计划都惩罚持续占用计算资源的行为。短连接虽然牺牲了实时性的极致，但换来了稳定、可预测的零成本运营。

我们把推送和拉取的间隔精确设为 5 秒，节点列表这种非实时请求拉长到 1 分钟。包体结构也做了优化——按 type 和 name 聚合条目，减少重复键名，体积缩小了 23.5%。精确的计算结果和节点工时数据，见“设计模型与实测数据”章节。免费账户不需要绑定信用卡。

</details>

## nexus 算法的数学模型与实测数据

我们不是在纸上谈兵。nexus 滑动窗口算法在真实跨国网络中经过了超过 12,000 个数据包的连续压测，运行时长超过 17 小时。

### 压测环境与条件

- **压测时长**：17.12 小时
- **总发送数据包**：12,223 个
- **网络环境**：跨国公网直连 Cloudflare Workers（无 VPN，无专线）
- **滑动窗口**：3 批次（T, T-1, T-2）

### 核心数据

| 指标 | 数值 | 计算方式 | 说明的问题 |
| :--- | :--- | :--- | :--- |
| **HTTP 连接失败率** | 15.05% | (总发送数 - 一次抵达成功数) ÷ 总发送数 × 100% | 公网裸传输的真实损耗——包括随机丢包和因网络审查导致的连接中断。每 100 次推送中约 15 次在第一次尝试时未能成功建立 HTTP 连接或完成数据传输 |
| **nexus 算法后最终连通率** | 98.25% | 至少抵达一次数 ÷ 总发送数 × 100% | 经过滑动窗口和Jitter Buffer冗余保护后，用户实际能看到的有效数据比例 |
| **二次抵达成功数** | 1,624 | 在第二次推送窗口（T-1）中被观测到的抵达记录 | 滑动窗口补偿机制在第二次推送中挽回的数据包数量——这些包在第一次推送时未能成功抵达，但被 T-1 批次救回 |
| **三次抵达成功数** | 1 | 在第三次推送窗口（T-2）中被观测到的抵达记录 | 极少数被两次连续失败后，仍被 T-2 批次救回的数据包。 |
| **网络中断丢失数** | 214 | 从未抵达的序列号数 | 这部分包含了因网络完全中断导致的连续失败，也包括连续三次 HTTP 连接均失败的情况。这部分数据中同时存在网络断层和连续三次丢包，而这两者正是我们算法目前的能力局限所在 |

### 指标解读

**HTTP 连接失败率 15.05%**：这是真实公网的质量。每 100 次推送中，大约 15 次在第一次尝试时无法成功建立 HTTP 连接或完成数据传输。失败的原因可能是随机丢包，也可能是网络审查导致的 TLS 握手阻断——我们的压测数据无法区分这两者，但它们都表现为 HTTP 连接失败。

**nexus 算法后最终连通率 98.25%**：这是用户实际看到的可靠性。经过三批次滑动窗口冗余保护后，98.25% 的数据包至少抵达了一次。剩下的 1.75% 是 214 个包。

**二次抵达成功数 1,624**：这 1,624 个包是 nexus 算法的核心价值。它们第一次推送时未能成功抵达，但被第二次推送（T-1 批次）成功送达。如果没有滑动窗口，这些数据将直接丢失，最终连通率会从 98.25% 掉到 84.95%——差了 13 个百分点。

**三次抵达成功数 1**：只有 1 个包在连续两次失败后，被 T-2 批次救回。这个数字极小，但恰好说明了两点：一是 HTTP 连接失败率确实在 15% 左右（三次都失败的概率理论上只有 0.34%，实测更低）；二是滑动窗口的第三层冗余在绝大多数情况下用不到，但它确实存在，并且在你不知道的时候可能已经工作过了。

**网络中断丢失数 214**：这是 nexus 算法目前的短板。我们不回避这一点——这 214 个包可能遭遇了网络完全中断，也可能遭遇了连续三次 HTTP 连接均失败。

### 抵达模型

我们在设计时预估，首次推送成功率为 95%，经过三次冗余后的最终失败率约为 0.05³ = 0.0125% 99.99%。本次压测中，实际首次成功率为 84.95%，理论失败率为0.1505³ = 0.034% 实际最终成功率为 98.25%。虽然观测值和理论值有差异，但是考虑到恶劣的工况、公网的不稳定性以及实际的用途，也就是仪表盘，我们的算法仍然能够在大多数时间保证看到数据。我们必须承认网络的复杂性，毕竟公网没有承诺每一百次http请求均匀的失败15次。

## 额度模型与实测数据

“免费”不是一个营销口号，而是一个经过精确计算和真实压测验证的工程承诺。以下是基于 12 小时三节点稳定运行的实测数据。

### 真实额度消耗

在 12 小时的三节点（一个前端、两个后端）稳定运行中：

- **最后一小时请求数**：1,991 次
- **12 小时总请求数**：23,930 次
- **单节点每小时平均消耗**：约 665 次
- **单节点全天消耗**：约 15,960 次
- **单账户每日免费额度**：100,000 次


### 理论额度消耗
- **每工时前端理论消耗**: 720次状态轮询 + 60次节点轮询 = 780次请求 + 切换节点操作次数
- **每工时后端理论消耗**: 720次状态轮推 = 720次请求
- **120节点工时设计指标下最高消耗** = 100000 / 120 = 833次请求
- **前端冗余量** = 833次请求 - 780次请求 = 53次请求
- **后端冗余量** = 833次请求 - 720次请求 = 113次请求

计算显示，我们每个前端理论消耗为780次请求 + 切换节点操作次数，每个后端理论消耗为720次请求。冗余出的168次请求足以覆盖每分钟三次的切换节点操作。而考虑到请求失败率，我们的冗余会更加可靠。

### 容量评估

一个免费账户足以支撑理论设计下的120节点工时，冗余量约为10%。这个余量足够覆盖临时流量波动、误操作、节点切换操作和额外的节点轮询——而且还能偶尔支持额外的查看需求，只不过存在潜在的超额风险。
